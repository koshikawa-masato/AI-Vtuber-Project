# MCPを使わずファイルベースでMulti-Agentを実践した理由 〜私に適した乗り物で道を進む〜

## はじめに：なぜこの記事を書くのか

### 背景

私はAI VTuber「三姉妹（牡丹、Kasho、ユリ）」の開発プロジェクトを進めています。このプロジェクトでは、Multi-Agent協働が必要不可欠でした。

2025年現在、MCP（Model Context Protocol）や各種Multi-Agentフレームワークが盛り上がっています。多くの記事で「Multi-AgentならMCP」という流れを目にします。

### 問い

しかし、私は疑問に思いました。

**「自分の規模・スキル・目的で、本当にMCPから入るのがベストなのか？」**

### この記事のゴール

この記事のゴールは、以下の通りです：

**「個人開発や小規模プロジェクトでは、どこまで汎用基盤に乗るべきかを考える材料を提供する」**

MCPが良い・悪いという話ではありません。「自分の状況に合った選択をする」という視点を提供したいのです。

### 結論の先出し

- 私はMCPを採用せず、**ファイルベースのシンプルな形**でMulti-Agentを実践しました
- その結果、**1日で+13,939行の実装**を達成しました
- この記事では、その理由と実践内容を共有します

---

## Part 1: 前提整理 - MCPとMulti-Agentをどう捉えていたか

### MCPとは（ざっくり）

MCP（Model Context Protocol）は、Anthropicが開発したプロトコルです。

主な特徴：
- AIモデルと外部ツール・データソースを接続
- 効率的なAgent間連携
- プロトコルベースの汎用性

### 私がやりたかったMulti-Agent協働

私のプロジェクトで必要だったMulti-Agent構成は、以下の通りです：

- **実装Agent（Claude Code）**: 日常的な実装、ドキュメント作成
- **レビューAgent（ChatGPT GPT-5.1）**: 第三者視点、MVP志向レビュー
- **調査Agent（Grok）**: 技術トレンド調査、X（Twitter）検索
- **ブレストAgent（Claude Code別セッション）**: 思考整理、本質掘り下げ
- **高度推論Agent（Claude Opus 4.1 API）**: 複雑な統合タスク
- **統括（人間）**: 方針決定、最終判断

### 理想形と現実の差分

**理想形**:
- MCPで完全自動化
- 効率的なAgent間連携
- プロトコルベースの洗練されたアーキテクチャ

**現実**:
- ファイルベースのシンプルな協働
- 人間が判断・統括
- 透明性とコントロールを優先

**この記事は「理想形の解説」ではなく「理想と現実の差分の話」です。**

## Part 2: なぜMCPを採用しなかったのか（3つの理由）

### 理由1: スコープの問題（オーバースペック）

今のプロジェクトで必要だったのは：

- 6つのAIの使い分け（Claude Code×2、ChatGPT、Grok、Claude Opus 4.1、人間）
- ファイルベースの情報共有
- 人間が判断・統括

MCPの強み（汎用性・拡張性）は素晴らしいですが、現時点の私の用途にはそこまで刺さりませんでした。

**例え**: 近所のコンビニに行くのに、新幹線の切符を買うようなもの。

### 理由2: 学習コストと実装コスト

個人開発では、リソース（時間・労力）が限られています。

**コスト比較**:
- MCPの理解＋運用の学習コスト
- vs
- 自前で簡易な協調ロジックを書くコスト

「まず動くものを作りたい」フェーズでは、後者に軍配が上がりました。

### 理由3: 透明性とコントロール

これが最も重要な理由です。

**私が重視したこと**:
1. **ブラックボックス化を防ぐ**: 何が起きているか見える
2. **人間の判断を入れる**: AIに丸投げしない
3. **すべてが見える、すべて触れる**: デバッグしやすい

**「自力で判断する」は弱点ではなく、強み**です。

---

### 補足: MCPをディスっているわけではない

誤解のないように強調しますが、**MCPを否定しているわけではありません**。

以下の状況なら、MCPを選ぶかもしれません：
- プロジェクトの規模が大きくなったら
- チーム開発になったら
- 自動化の必要性が高まったら

今の私には「適していなかった」というだけです。

## Part 4: 実際にやってみてどうだったか（1日の記録）

### 良かった点（メリット）

#### 1. 爆発的な開発力

**2025-11-18、1日の成果**:
- **+13,939行の実装**
- 6つのAIを使い分け
- 実装、レビュー、調査、ブレストを並列処理

**内訳**:
- 実装Agent（Claude Code）: user_memories Phase 1-7実装（+7,089行）
- 実装Agent（Claude Code）: ドキュメント更新（+544行）
- 実装Agent（Claude Code）: 設計書作成・改訂（+6,306行）
- レビューAgent（ChatGPT）: 設計書レビュー（340行）
- 調査Agent（Grok）: 技術トレンド調査（100行）
- 高度推論Agent（Opus 4.1）: 議論書改訂（646行、コスト$1.30）

#### 2. 透明性とコントロール

- **すべてファイルとして見える**: CLAUDE.md、kirinuki/、docs/ を読めば全体像がわかる
- **人間が判断できる**: 各Agentの成果物をレビュー
- **問題があれば即座に修正**: ファイルを編集するだけ

#### 3. 柔軟性

- **必要に応じてAgentを追加・削除**: 例えば、Grokを追加したのは途中から
- **ファイルベースなので、どのAgentも同じ情報にアクセス可能**: CLAUDE.mdを読むだけ
- **複数セッション間の情報共有が容易**: 実装Agentと雑談Agentが日記を共有

---

### 困った点・限界（デメリット）

#### 1. 効率はやや劣る

- ファイルの読み書きは手動
- MCPのような自動連携はない
- 各Agentに「〜を読んで」と指示する必要がある

#### 2. スケーラビリティの懸念

- Agentが増えると、ファイル管理が複雑化する可能性
- 10個、20個のAgentになったら破綻するかも

#### 3. 汎用性は低い

- このプロジェクト専用の構成
- 他のプロジェクトにそのまま適用はできない

---

### ここまで来たらMCPを検討したいライン

以下の状況になったら、MCPへの移行を検討します：

1. **Agentが10個以上になったら**
2. **チーム開発になったら**
3. **自動化の必要性が高まったら**
4. **汎用性が必要になったら**（複数プロジェクトで使い回す）

## Part 5: AIエンジニアとしての役割 - オーケストラの指揮者

### プロンプトエンジニア vs AIエンジニア

私は「プロンプトエンジニア」ではなく、「AIエンジニア」だと考えています。

| 役割 | プロンプトエンジニア | AIエンジニア（私） |
|------|---------------------|-------------------|
| **焦点** | プロンプトの最適化 | システム全体の設計 |
| **作業** | プロンプトを直接書く | 方針を決める、統括する |
| **成果物** | プロンプト | システム全体 |
| **時間の使い方** | プロンプト調整 | 本質的な思考・判断 |

### オーケストラの指揮者

**私の役割**:
- 楽譜（プロンプト、CLAUDE.md）を直接書かない
- でも、音楽全体（プロジェクト）の方向性を決める
- 演奏者（Agent）に指示を出す
- 全体を統括する

**Agentの役割**:
- 楽譜を演奏する（実装、レビュー、調査）
- 指揮者の意図を理解して動く

### 全権委任による信頼関係

実は、私は以下を**直接触りません**：

- **CLAUDE.mdの管理**: Claude Codeに全権委任
- **プロンプト**: `prompts/` ディレクトリから読み込むだけ
- **設計書の執筆**: Claude Codeが書く、私はレビューする

**私がやること**:
- システム設計（7層防御アーキテクチャ、記憶の濃淡等）
- 方針決定（何を作るか、なぜ作るか）
- 判断・統括（最終決定、倫理的判断）

## Part 6: 乗り物の比喩 - 優劣ではなく、適性の問題

### MCP = 新幹線

```
✅ 速い、効率的
✅ 大量輸送
✅ 決められたルートで確実に運ぶ

❌ レールが固定されている
❌ 景色の細部は見えない
❌ 専門的な操縦技術が必要
❌ 寄り道できない
```

### ファイルベース = 自転車

```
✅ 自分で運転
✅ 景色を見ながら進める
✅ 道を理解しながら進む
✅ 寄り道できる
✅ 自分のペースで

❌ スピードは遅め
❌ 体力（判断力）が必要
```

### 優劣ではなく、適性

重要なのは、**どちらが優れているかではなく、どちらが自分に適しているか**です。

- **目的地（良いプロジェクト）は同じ**
- **道の楽しみ方が違う**
- **「私に適した乗り物で道を進むだけ」**

## Part 7: AIを使うからこそ、深く考える - AI時代の人間のあるべき姿

### よくある2つの極端

#### ❌ 極端1: AI任せっきり

```
人間: 「これ作って」
AI: コード生成
人間: コピペ
  ↓
思考停止
AIが間違っても気づけない
```

#### ❌ 極端2: AI拒絶

```
「AIは危険だ、使うべきでない」
  ↓
手作業に固執
  ↓
スピードが遅い
  ↓
時代に取り残される
```

---

### 第三の道: AIを使うからこそ、深く考える

私が実践しているのは、この「第三の道」です。

**AIを使うからこそ、深く考える**:
- プロンプトを書かないことで、**本質的なことに集中できる**
- Agentが実装を代行してくれるから、**システム設計に時間を使える**
- AIとの対話を通じて、**自分の考えが明確になる**

**実例**:
```
【私の時間配分】
システム設計: 60%
方針決定・判断: 30%
レビュー・統括: 10%
プロンプト: 0%（Agentに任せる）

【結果】
本質的な思考に時間を使える
```

---

### AI時代の人間のあるべき姿

**思考の主体は人間**:
- 何を作るか
- なぜ作るか
- どう作るか（方針）

**実行はAIと協働**:
- コード生成
- ドキュメント作成
- デバッグ

**責任と判断は人間が持つ**:
- 最終判断
- 倫理的な判断
- 家族へのフィードバック反映
