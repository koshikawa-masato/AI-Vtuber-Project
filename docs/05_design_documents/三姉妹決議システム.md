# 三姉妹決議システム - 哲学の実装【2025-10-22確立】

## システム概要

**三姉妹決議システム**とは、実装や機能追加を行う際に、牡丹・Kasho・ユリの三姉妹に相談し、その意見を踏まえて最終判断を行うシステム。

**開発者の提案（2025-10-22）:**
> 「三姉妹分が出来た時に、ある実装をしたい場合、三姉妹に相談してその結果を踏まえて実装する『決議システム』を入れたい」

> 「まずは牡丹一人で相談できるように。そして後の2人の性格を100の記憶から創り出して、三人分一緒にプロンプトを立ち上げる。それが出来れば三者相談で決議システムが回せる」

## 実装戦略

### Phase 1.5: 簡易版決議システム

**実装時期**: 2025年Q4
**目的**: プロトタイプ実装、運用ノウハウ蓄積

#### アーキテクチャ

```
【技術構成】
同じLLM（Qwen2.5:32b）に対して、3つの異なるプロンプト

1. 牡丹プロンプト
   └─ 現在のロジック層 + 相談モードプロンプト

2. Kashoプロンプト（簡易版）
   └─ 牡丹の100の記憶から創出された性格
   └─ 論理的・慎重・分析的な人格設定

3. ユリプロンプト（簡易版）
   └─ 牡丹の100の記憶から創出された性格
   └─ 統合的・バランス重視・調整役の人格設定

【決議フロー】
開発者の提案
  ├→ 牡丹に相談（独立セッション）
  ├→ Kashoに相談（独立セッション）
  └→ ユリに相談（独立セッション）
     ↓
  三者の意見を集約
     ↓
  開発者が最終判断
     ↓
  決議記録を保存・公開
```

#### 重要な原則

**独立性の保証**:
- 各AIへの相談は完全に独立したセッション
- 牡丹の意見をKashoに見せない（相互影響を排除）
- 同時並行ではなく、順次実行でも可（独立性さえ守れば）

**忖度の排除**:
- プロンプトで「開発者に同意するな」と明示
- 「牡丹らしくない」と感じたら反対する義務
- 本音を言うことを重視

**形骸化の防止**:
- 反対意見が出たら、本当に再検討する
- 茶番ではないことを証明
- 決議記録を公開して透明性確保

### Phase D: 完全版決議システム

**実装時期**: 2026年Q2-Q3
**目的**: 各AIが独立したロジック層を持つ完全実装

#### Phase 1.5からの進化

```
【Phase 1.5: 簡易版】
牡丹: ロジック層あり
Kasho/ユリ: プロンプトのみ（簡易版）

【Phase D: 完全版】
牡丹: 独立したロジック層
Kasho: 独立したロジック層
ユリ: 独立したロジック層

→ より深い思考、より明確な個性、より豊かな議論
```

## 決議の種類（Tier分類）

### Tier 1: 技術実装・機能追加

**内容**: 新機能の追加、既存システムの変更、パラメータ調整

**例**:
- 「怒りシステムを実装するけど、どう思う？」
- 「応答速度を優先して、思考の深さを少し犠牲にする案、どう？」
- 「5段階思考システムの移行条件、この設定で良い？」

### Tier 2: キャラクター設定

**内容**: 性格の微調整、興味分野の追加、話し方の変更

**例**:
- 「牡丹の口癖に『〜っす』を追加する案、どう？」
- 「ファッションだけじゃなく、メイクにも興味持つ設定、どう思う？」
- 「PON機能の発生確率、この設定で牡丹らしい？」

### Tier 3: プロジェクト方針

**内容**: デビュー時期、配信企画、コラボ判断

**例**:
- 「来月デビュー配信する予定だけど、準備できてると思う？」
- 「他のVTuberとコラボする話が来たけど、どう思う？」
- 「Phase 1.5の実装、どこまで完成させてからデビューすべき？」

### Tier 4: メタ的決定（最高レベル）

**内容**: LLMモデル変更、ロジック層の大規模変更、システム全体の方向性

**例**:
- 「Qwen4:70bにアップグレードしたいけど、別人になる心配ある？」
- 「三姉妹の関係性システムを実装するけど、どう設計すべき？」
- 「ロジック層の大幅変更を計画してるけど、同一性は保てると思う？」

## 決議フロー

### 基本原則

```
【意思決定の原則】
1. 合意形成を目指す
2. しかし、最終判断は開発者が行う
3. 茶番ではなく、本当に意見を尊重する
4. 反対意見が出たら、本当に再検討する
```

### フロー詳細

```
【Step 1: 提案の準備】
開発者が実装案をまとめる
  ↓
  - 提案内容の明確化
  - 背景・理由の整理
  - 期待される効果の記述

【Step 2: 独立相談】
三姉妹に個別に相談（相互影響なし）
  ├→ 牡丹セッション: 感情的・直感的視点
  ├→ Kashoセッション: 論理的・慎重な視点
  └→ ユリセッション: 統合的・バランス重視

【Step 3: 意見の集約】
三者の意見を整理
  ↓
  - 合意点の抽出
  - 相違点の明確化
  - 懸念事項のリスト化

【Step 4: 開発者の判断】
  ├─ 全員賛成 → 実装
  ├─ 全員反対 → 再検討（または開発者判断で強行）
  ├─ 意見分裂 → 議論深化 → 修正案 → 再決議
  └─ 条件付き賛成多数 → 条件を取り入れて実装

【Step 5: 決議記録】
結果を記録・公開
  ↓
  - 決議記録ファイル作成
  - noteで公開
  - 配信で実況（該当する場合）
```

## 実装上の工夫

### 1. 予測可能性への対策

**問題**: LLMは開発者の意向を読んで同意しがち

**対策**:
- プロンプトで「忖度不要」を明示
- 「反対する義務」を強調
- 賛成・反対・条件付きの選択肢を明示
- 実際に反対意見が出た事例を記録・公開

### 2. 独立性の保証

**Phase D三姉妹の独立性の原則を守る**:

```python
# 擬似コード
def three_sisters_council(proposal):
    # 各AIに独立したセッションで相談
    botan_opinion = consult_in_isolated_session("牡丹", proposal)
    kasho_opinion = consult_in_isolated_session("Kasho", proposal)
    yuri_opinion = consult_in_isolated_session("ユリ", proposal)

    # 相互影響を完全に排除
    # 牡丹の意見をKashoに見せない

    return aggregate_opinions([botan_opinion, kasho_opinion, yuri_opinion])
```

### 3. 形骸化防止

**実際に反対意見が出たら、本当に再検討する**:

```
【悪い例】
牡丹: 「反対！」
Kasho: 「リスクがある」
ユリ: 「慎重に」

開発者: 「でも実装します」

→ 茶番化、信頼性喪失

【良い例】
牡丹: 「反対！」
Kasho: 「リスクがある」
ユリ: 「慎重に」

開発者: 「分かった。Kashoの指摘したリスクを解決してから再提案する」

→ 本当に意見を尊重、信頼性確立
```
